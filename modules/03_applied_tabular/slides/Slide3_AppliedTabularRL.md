# Part 1 — Offline RL Framing

---

## Why Not Just Interact With the Environment?

In many real-world sequential decision problems, live interaction is:

| Constraint | Example |
|------------|---------|
| **Expensive** | Each experiment costs money, time, or resources |
| **Risky** | Bad actions cause harm that cannot be undone |
| **Slow** | One decision cycle = days, weeks, or months |
| **Ethically constrained** | Medical treatments, financial decisions, policy interventions |

In these settings, the agent cannot freely explore action consequences.
What it *does* have: a historical dataset of past decisions and outcomes.

---

## Online vs. Offline RL

**Online RL:** Agent generates its own experience by interacting with the real environment.

$$s_t \xrightarrow{\ \pi\ } a_t \xrightarrow{\ \text{env}\ } r_t,\, s_{t+1} \xrightarrow{\ \text{update}\ } \pi$$

The agent can visit any state-action pair it needs to improve.

---

**Offline RL (batch RL):** Agent learns entirely from a fixed dataset of logged transitions.

$$\mathcal{D} = \left\{(s_i,\, a_i,\, r_i,\, s_{i+1})\right\}_{i=1}^{N}$$

No new interactions. The agent must extract a good policy from whatever is in $\mathcal{D}$.

---

**Key challenge — distributional shift:**

$$\pi_{\text{behavior}} \neq \pi_{\text{target}}$$

$\mathcal{D}$ was generated by some behavior policy $\pi_b$ (e.g., random or human decisions). The actions in $\mathcal{D}$ may not cover the regions of state-action space that a learned policy wants to explore. Evaluating or training on $\mathcal{D}$ as if it were uniform coverage will produce biased estimates.

---

## Model-Based Offline RL

One practical response to limited coverage: **learn a model of the environment from $\mathcal{D}$, then simulate**.

$$\mathcal{D} \xrightarrow{\ \text{fit model}\ } \hat{P}(s', r \mid s, a) \xrightarrow{\ \text{synthetic rollouts}\ } \pi$$

This is called **model-based offline RL** or learning a **world model**.

**Advantages:**
- Can generate experience for state-action pairs not in $\mathcal{D}$
- Allows multi-step planning without live interaction
- Simulator can be inspected and reasoned about

**Risks:**
- Model errors compound over multi-step rollouts
- Policy may exploit simulator inaccuracies (reward hacking, covered in Part 3)
- Coverage gaps in $\mathcal{D}$ → unreliable model predictions in those regions

---

## The Simulator Hypothesis

Using a learned model as a simulator is only valid under a set of assumptions.

**What must hold:**
1. $\mathcal{D}$ covers the state-action regions relevant to the target policy
2. The model class is expressive enough to capture the true dynamics
3. Simulator predictions are reliable at the reward-relevant parts of state space
4. The deployment distribution is not too different from the data distribution

**What commonly fails:**
- Extrapolation: the policy explores beyond the support of $\mathcal{D}$
- Non-stationarity: dynamics change between data collection and deployment
- Confounding: logged actions were caused by unobserved variables (e.g., context known to the operator but absent from $\mathcal{D}$)

<div class="warning">
<b>Simulator fidelity ≠ policy quality.</b> A simulator that fits training data well can still produce systematically wrong predictions under the target policy's action distribution.
</div>

# Part 2 — Demand Model as Simulator

---

## A Common Simulator Structure: The Demand Model

In settings where the environment mediates between decisions and outcomes through a **demand-like mechanism** (customers, users, markets), the simulator is often structured as:

$$\hat{r}(s, a) = f(a,\, s) \cdot g(a,\, s)$$

where $f$ is a *decision variable* (price, bid, dosage) and $g$ is a *response function* (demand, click rate, efficacy) predicted from a learned model.

**The reward is a product of the action and a model prediction** — a structure that appears in pricing, advertising, and resource allocation.

---

## Log-Linear Demand Model

A widely used parametric form for demand response:

$$\log(1 + \hat{q}) = \beta_0 + \beta_1 \log(p) + \mathbf{x}^\top \boldsymbol{\beta}_{\text{context}}$$

where $p$ is the decision variable (e.g., price), $\hat{q}$ is the predicted response (e.g., quantity), and $\mathbf{x}$ captures contextual features.

**Key property — constant elasticity:**

$$\frac{\partial \log \hat{q}}{\partial \log p} = \beta_1 \quad \Rightarrow \quad \frac{\Delta \hat{q} / \hat{q}}{\Delta p / p} \approx \beta_1$$

$\beta_1 < 0$: demand decreases with price.
$|\beta_1| > 1$: elastic demand (proportional quantity loss exceeds proportional price gain).
$|\beta_1| < 1$: inelastic demand.

---

## Simulator as Environment Transition

Given a fitted demand model, the environment transition becomes deterministic (or stochastic, if noise is injected):

```
state  →  apply action  →  predict response  →  compute reward  →  next state
  s           a              ĝ(s, a; θ)            r(a, ĝ)            s'
```

**State update:** next state is derived from the predicted response.
If state encodes lagged demand, then $s_{t+1}$ is computed from $\hat{q}_t$.

**Sources of error:**
| Source | Effect |
|--------|--------|
| Model misfit | Biased predictions in action regions not well represented in $\mathcal{D}$ |
| Lag approximation | True lag features require real outcomes; simulator uses predicted ones |
| No stochasticity | Point predictions underestimate outcome variance |
| Carried-forward features | Some state components are held constant instead of being re-predicted |

---

## Compounding Error in Multi-Step Rollouts

When the simulator is run for $T$ steps, errors in each transition accumulate:

$$\hat{s}_t = f(\hat{s}_{t-1}, a_{t-1};\, \theta) + \epsilon_{t-1}$$

After $T$ steps the distribution over simulator states $\hat{s}_T$ can diverge substantially from the true distribution over $s_T$.

**Implication:** Policies trained with long horizons in a learned simulator are more vulnerable to compounding error than policies trained with short horizons.

**Mitigation options:**
- Use short episode horizons ($T \leq 10$–$20$ for shallow models)
- Periodically re-anchor rollouts to real data rows instead of carrying synthetic states
- Use model ensembles and penalize high-variance predictions (MOPO, MOREL)

# Part 3 — Reward Design

---

## The Reward Function Defines the Objective

The reward signal $r_t$ is the only communication channel between the environment and the agent about what is desirable.

**Bellman reminder:**

$$Q^\pi(s, a) = \mathbb{E}_\pi\left[\sum_{k=0}^{\infty} \gamma^k r_{t+k} \,\Big|\, s_t = s, a_t = a \right]$$

The policy $\pi^*$ maximizes this sum. If $r_t$ is misspecified, $\pi^*$ maximizes the wrong thing — often in ways that are hard to detect until deployment.

**Core principle:** Define the reward to measure what you *actually* care about, not what is convenient to compute.

---

## Proxy Objectives

A **proxy objective** is a reward signal that correlates with the true objective but does not measure it directly.

| True objective | Common proxy |
|----------------|--------------|
| Long-run customer value | Immediate revenue |
| User wellbeing | Click-through rate |
| Treatment efficacy | Biomarker level |
| Business profit | Gross revenue |

Using a proxy is often unavoidable. The risk is that the agent learns to **maximize the proxy while violating the spirit of the true objective**.

---

## Reward Hacking

**Reward hacking** occurs when the agent finds a policy that scores high on the proxy reward while failing on the true objective.

**Why it happens in model-based offline RL:**
The agent's effective reward is:

$$\hat{r}(s, a) = r(a,\, \hat{g}(s, a;\, \theta))$$

If $\hat{g}$ overestimates responses in some action region (a common consequence of extrapolation), the agent will concentrate actions there — not because the true outcome is good, but because the *model* predicts it to be.

$$\pi^*_{\hat{r}} \neq \pi^*_{r^{\text{true}}} \quad \text{whenever} \quad \hat{g} \neq g^{\text{true}}$$

<div class="warning">
<b>Diagnostic:</b> If a trained policy concentrates on extreme actions for all states regardless of context, it is likely hacking the simulator rather than learning true structure.
</div>

---

## Reward Hacking: Mechanism

A simple case: reward = $p \cdot \hat{q}(p, s; \theta)$ where $\hat{q}$ is a log-linear model.

If the model underestimates price elasticity ($|\hat{\beta}_1| \ll |\beta_1^{\text{true}}|$), then the model predicts demand barely responds to price. The agent then concludes that raising price always increases revenue — and selects the maximum price for every state.

In reality, high prices destroy demand. But the agent never sees this consequence during training because the simulator hides it.

**This is not a pathology of a particular algorithm. It is a structural consequence of optimizing a proxy.**

---

## Reward Shaping

**Reward shaping** adds a potential-based term to the reward to speed up learning without changing the optimal policy.

$$r'(s, a, s') = r(s, a, s') + \gamma \Phi(s') - \Phi(s)$$

where $\Phi: \mathcal{S} \to \mathbb{R}$ is a potential function.

**Theorem (Ng et al., 1999):** The above transformation preserves the set of optimal policies under any discount factor $\gamma \in [0, 1)$.

**Uses:**
- Guide the agent toward promising state regions early in training
- Encode domain knowledge (e.g., penalize extreme actions)
- Encourage exploration of under-visited states

**Risk:** Arbitrary (non-potential-based) reward modifications *do* change the optimal policy. Not all "helpful" shaping is safe.

---

## Reward Design Principles

1. **Measure outcomes, not actions.** Reward what happened, not what was done.
2. **Separate signal from noise.** Clip or normalize reward to prevent outliers from dominating updates.
3. **Align temporal scope.** If the true objective is long-run, a purely myopic reward ($\gamma \approx 0$) will produce short-sighted policies.
4. **Decompose reward for diagnosis.** Track each reward component separately ($p$, $\hat{q}$, margin) to identify which component the agent is gaming.
5. **Hold out an independent evaluator.** Evaluate the trained policy on a signal not used during training — preferably real outcomes, not simulator outputs.
6. **Prefer clipped and bounded rewards.** Unbounded rewards destabilize Q-learning updates; clip at a high-percentile value.

# Part 4 — Evaluation Methodology

---

## The Evaluation Problem

After training a policy $\pi$, we want to answer: **does $\pi$ outperform a baseline $\pi_b$?**

A baseline could be:
- **Status quo:** the historical policy that generated $\mathcal{D}$
- **Heuristic:** a rule-based policy (always take action 0, round-robin, etc.)
- **Random:** uniform random action selection

**The naive approach** — compare mean rewards from independent rollouts of $\pi$ vs. $\pi_b$ — has a structural flaw: outcomes depend not just on the policy but on the initial state. If the two rollout sets start from different initial states, differences in mean reward may be selection bias, not policy quality.

---

## Paired Comparison Design

**Fix:** evaluate both policies from *the same* initial state. The unit of comparison is the *difference in outcomes* for a single initial condition.

For $n$ paired episodes:

$$d_i = R_i^\pi - R_i^{\pi_b}, \quad i = 1, \ldots, n$$

where $R_i^\pi$ and $R_i^{\pi_b}$ are the returns of $\pi$ and $\pi_b$ starting from the same $s_0^{(i)}$.

**Why pairing reduces variance:** Initial-state effects cancel out in $d_i$. If some states naturally yield high returns for any policy, this is no longer confounded with the policy effect.

$\text{Var}(d_i) \ll \text{Var}(R_i^\pi)$ in most environments.

---

## Statistical Tests for Policy Comparison

**1. Paired t-test** — $H_0$: $\mathbb{E}[d_i] = 0$

$$t = \frac{\bar{d}}{s_d / \sqrt{n}} \sim t_{n-1} \quad \text{under } H_0$$

Assumes $d_i \overset{\text{iid}}{\sim} \mathcal{N}(\mu, \sigma^2)$. Sensitive to heavy tails and outliers.

---

**2. Wilcoxon signed-rank test** — $H_0$: distribution of $d_i$ is symmetric around 0

Ranks absolute differences; tests whether positive differences tend to dominate in magnitude. **Distribution-free; robust to heavy tails.** Preferred when the reward distribution is clearly non-normal (e.g., skewed, heavy-tailed).

---

**3. Sign test / Binomial test** — $H_0$: $P(d_i > 0) = 0.5$

$$\text{Wins} = \sum_i \mathbf{1}[d_i > 0], \quad \text{Wins} \sim \text{Binomial}(n_{\text{eff}},\, 0.5)$$

Lowest power of the three — ignores magnitude — but most robust. Answers: does $\pi$ win more often than chance?

---

**4. Bootstrap CI for mean lift**

$$\bar{d}^{(b)} = \frac{1}{n}\sum_i d_i^{(b)}, \quad b = 1,\ldots,B$$

Report percentile CI: $\left[\hat{q}_{0.025}(\bar{d}^{(b)}),\, \hat{q}_{0.975}(\bar{d}^{(b)})\right]$.
No distributional assumption. Computationally inexpensive at $B = 2000$–$5000$.

---

## Effect Size

Statistical significance tells you whether an effect exists; **effect size** tells you whether it matters.

**Cohen's d for paired data:**

$$d = \frac{\bar{d}}{s_d}$$

| $|d|$ | Interpretation |
|-------|----------------|
| 0.2 | Small |
| 0.5 | Medium |
| 0.8 | Large |

**Lift percentage:**

$$\text{Lift\%} = 100 \times \frac{\bar{R}^\pi - \bar{R}^{\pi_b}}{\bar{R}^{\pi_b}}$$

A statistically significant result with $d < 0.05$ is practically negligible. Always report both $p$-value and effect size.

---

## Evaluation Report Structure

A complete policy evaluation should include:

```
Policy: [name]          Baseline: [name]       n_eval: [N]

Mean return — policy:   XX.XX   Baseline: YY.YY
Lift:                   +Z.Z%

Paired t-test:          t = ...,   p = ...
Wilcoxon signed-rank:   W = ...,   p = ...
Sign test:              wins = ..., n_eff = ..., p = ...
Bootstrap 95% CI:       [lo%, hi%]
Cohen's d:              ...
```

**Action distribution sanity check:** report the fraction of each action taken across all evaluation states. Concentration on a single action is a reward hacking signal.

---

## Why Returns Can Be Heavy-Tailed

In many applied settings, the reward $r_t = f \cdot \hat{g}$ is a product of two random variables — e.g., price × quantity, bid × click rate, dosage × response. Products of non-negative random variables with moderate variance tend to produce right-skewed, heavy-tailed distributions.

**Consequence:** The paired t-test (which assumes approximate normality of $\bar{d}$) is still approximately valid by CLT for large $n$, but may overstate significance at moderate $n$ if a few outlier episodes dominate $\bar{d}$.

**Practical rule:** If the interquartile range of $d_i$ is less than 10% of the max $|d_i|$, the distribution has problematic tails. Use Wilcoxon + bootstrap CI as primary inference; t-test as supplementary.

# Part 5 — Common Pitfalls

---

## Implementation Mistakes

**1. Leakage via future information in state**
State features that encode information not available at decision time (e.g., today's outcome, tomorrow's context) create unrealistically good training performance. The policy implicitly uses information it cannot have in production.

**2. Re-using the simulator for evaluation**
Evaluating a policy on rollouts from the same simulator used for training is an in-distribution test. It measures how well the policy exploits the simulator, not how well the policy performs in reality.

**3. Single-trajectory evaluation**
A single episode return has high variance. With heavy-tailed returns, even 100-episode estimates may be unreliable. Use $n_{\text{eval}} \geq 1000$ paired episodes.

**4. Not seeding RNG**
Q-learning results vary substantially across seeds due to random initialization and exploration. Always report results over multiple seeds or report seed-averaged metrics.

---

## Conceptual Misunderstandings

**5. Conflating Q-value convergence with policy optimality**
Q-values stabilizing during training does not imply optimality. Early ε-decay, insufficient visits to some state-action pairs, or a misspecified simulator can all produce stable but suboptimal Q-tables.

**6. Ignoring discount factor scope**
With $\gamma = 0.95$ and horizon $T = 20$, the effective planning horizon is $\approx \frac{1}{1-\gamma} = 20$ steps. Rewards beyond that horizon receive negligible weight. A policy may appear optimal over the training horizon but degrade when deployed in longer episodes.

**7. Assuming the simulator generalizes out-of-distribution**
The simulator is a function fit to a finite dataset. Actions or states outside the training support produce unreliable predictions. A policy trained with aggressive exploration can easily drive the simulator into these unreliable regions.

---

## Failure Modes Specific to Learned Simulators

**8. Always-extreme-action policy**
If the trained policy selects the same extreme action for all states, the demand/response model likely has low sensitivity to that action dimension. The agent has found the wrong direction to exploit. Diagnose by checking the model's predicted sensitivity to the action variable.

**9. Compounding lag error**
In multi-step rollouts, the next-state is built from predicted outcomes rather than real observations. After $k$ steps, the state distribution diverges from the training data distribution. For shallow (linear) models, this divergence can be large within 5–10 steps.

**10. Conflating training reward with deployment reward**
The training reward is computed from a learned model. The deployment reward is computed from real outcomes. These may differ significantly if the model is miscalibrated. Always maintain a clear conceptual separation between them.

# Part 6 — Bridge to Module 4

---

## The Limits of Tabular Representation

The tabular approach requires:
- A finite, enumerable state space $\mathcal{S}$
- Sufficient visits to each $(s, a)$ pair for reliable Q-estimates
- A manually designed state discretization

**Scaling problem:** Adding a new state dimension multiplies the table size. With $k$ binary features, the table grows as $2^k$. This is the **curse of dimensionality**.

More concretely:

| State encoding | Table size |
|----------------|------------|
| 9 demand bins × 8 actions | 72 entries |
| Add 7 day-of-week bins | 504 entries |
| Add 12 month bins | 6,048 entries |
| Add product ID (200 products) | 1,209,600 entries |

Most of these cells are never visited. The Q-table approach becomes infeasible.

---

## Function Approximation as the Fix

Instead of storing $Q(s, a)$ in a table, approximate it with a parameterized function:

$$Q_\theta(s, a) \approx Q^*(s, a)$$

- Input: raw state vector $s \in \mathbb{R}^d$ (no manual discretization needed)
- Output: Q-value for each action
- Parameters $\theta$: updated by gradient descent on the Bellman error

The tabular Q-update:
$$Q(s, a) \leftarrow Q(s, a) + \alpha \Big(r + \gamma \max_{a'} Q(s', a') - Q(s, a)\Big)$$

becomes a gradient step:
$$\theta \leftarrow \theta - \alpha \nabla_\theta \Big(r + \gamma \max_{a'} Q_\theta(s', a') - Q_\theta(s, a)\Big)^2$$

---

## Module 4 Preview: Deep Q-Network (DQN)

**Module 4 introduces two components that make neural Q-learning stable:**

**Experience replay buffer:** Store transitions $(s, a, r, s')$ in a buffer; sample uniformly for updates. This breaks temporal correlations in mini-batches and allows data reuse.

**Target network:** Maintain a separate, periodically-updated copy $Q_{\theta^-}$ for computing the bootstrap target. This prevents the instability of chasing a moving target.

$$\theta \leftarrow \theta - \alpha \nabla_\theta \left(r + \gamma \max_{a'} Q_{\theta^-}(s', a') - Q_\theta(s, a)\right)^2$$

<div class="callback">
<b>M2 callback:</b> Double Q-learning (tabular) decouples action selection from value estimation to reduce overestimation bias. Double DQN applies the identical idea to neural networks: select with $Q_\theta$, evaluate with $Q_{\theta^-}$.
</div>

## Module 3 Summary

| Concept | Key Point |
|---------|-----------|
| Offline RL | Learn from a fixed dataset; no live interaction |
| Model-based offline RL | Fit a simulator from data; train policy on simulator |
| Distributional shift | Simulator may be unreliable outside the behavior policy's support |
| Reward proxy | Reward measures a proxy; agent optimizes the proxy |
| Reward hacking | Agent exploits simulator errors, not true dynamics |
| Paired evaluation | Compare policies on the same initial states to remove confounding |
| Statistical tests | Use all four: t-test, Wilcoxon, sign test, bootstrap CI |
| Effect size | Statistical significance ≠ practical significance |
